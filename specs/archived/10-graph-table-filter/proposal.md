## 摘要

当前关系图（GraphView）展示项目下所有表的全部关系。项目规模较大时（30+ 表）图面拥挤，用户难以聚焦于自己关心的表及其关联。本提案增加按表名多选筛选的功能，让用户只查看所选表及其相互之间的关联关系。

## 动机

- 项目包含大量表时，关系图节点过多，视觉杂乱，难以定位目标表
- 现有 `type` 和 `min_confidence` 筛选只过滤关系（边），不影响节点（表），无法缩小表范围
- 用户经常只关心某几个业务模块相关的表（如 "order"、"order_item"、"product"），希望只看到这几个表及其关联
- 前端目前已有项目表列表页，用户期望在关系图页面也能复用"选择关注的表"的能力

## 范围

### 包含

- 后端 API 增加 `table_ids` 查询参数，接收表 ID 列表，过滤只返回指定表及其关联关系
- 后端逻辑：当指定 `table_ids` 时，只返回这些表之间的边（即边两端都在指定集合内）；同时节点只包含这些表；不指定时保持现有行为（返回全部）
- 前端 GraphView 增加表多选下拉/弹窗组件
- 表选项从项目表列表获取（复用现有接口）
- 两种视图（力导向图 / Mermaid ER 图）均受表筛选影响
- 筛选后显示已选表数量 + 总表数（如 "已选 5/20 个表"）

### 不包含

- 表选择与关系类型/置信度筛选的逻辑组合优化（"先选表再选关系"的 UX 引导）
- 保存用户筛选偏好到服务端
- 搜索不到表时的 "上传 SQL" 引导（项目本身上传后才进入关系图）

## 验收标准

- [ ] 不选择任何表（默认）: 关系图与现有行为一致，展示全部表及关系
- [ ] 选择部分表后: 图中只显示被选中的表节点，以及这些表之间的关联边
- [ ] 表选择组件支持搜索过滤表名，方便快速定位
- [ ] 表筛选与关系类型（FOREIGN_KEY / INFERRED）、置信度筛选协同工作，两者取交集
- [ ] 切换到 Mermaid 视图时表筛选同样生效
- [ ] 切换项目后表选择重置（不跨项目记忆）

## 技术方案概要

- 后端：`GET /api/projects/{id}/graph?table_ids=id1,id2,id3`，在 `build_graph` 中增加 table_ids 参数，传递给 Repository 或做内存过滤
- 后端：`GET /api/projects/{id}/mermaid` 同步增加 `table_ids` 参数
- 前端：封装 `TableSelect` 多选组件，请求 `/api/projects/{id}/tables` 获取表列表作为选项
- 前端：`useGraph` hook 增加 `tableIds` 参数，拼接到 query string
