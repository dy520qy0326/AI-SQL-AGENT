# PRD: 自然语言查询模块 (NL Query)

## 1. 产品概述

自然语言查询模块允许用户用自然语言询问关于数据库结构的问题，例如"用户表和订单表的关系是什么？"、"有哪些表用到了 user_id 字段？"。系统将表结构作为上下文注入，由 AI 理解并回答。

## 2. 目标用户

- **主要用户**：新加入项目的开发、数据分析师、需要快速了解数据结构的非技术角色
- **典型场景**：
  - "订单的总金额在哪个字段？"
  - "用户和地址是什么关系？一对一还是一对多？"
  - "列举所有包含 deleted_at 字段的表"
  - "从用户到订单的完整关联路径是什么？"

## 3. 功能需求

### P0 — 基础问答

| 功能 | 说明 |
|------|------|
| Schema 感知对话 | AI 理解整个项目的表结构上下文 |
| 字段查询 | "xxx 字段在哪张表？有哪些表有 xxx 字段？" |
| 关系查询 | "A 表和 B 表怎么关联的？" |
| 路径查询 | "从 A 表到 C 表经过哪些中间表？" |

### P1 — 增强对话

| 功能 | 说明 |
|------|------|
| 追问澄清 | 问题模糊时 AI 反问，"您指的是 user_id 还是 created_by？" |
| 上下文记忆 | 同一会话中后续问题基于前文理解 |
| 引用溯源 | AI 回答中标注信息来源（哪个表、哪个字段） |
| 建议问题 | 对话开始时 AI 推荐 3-5 个常用问题 |

### P2 — 高级能力

| 功能 | 说明 |
|------|------|
| SQL 生成 | "查询每个用户的订单总数" → 生成对应的 SQL |
| 差异分析 | "这个表和上一版本有什么不同？" |
| 模式推荐 | "这种设计是否是标准的星型模型？" |

## 4. 交互流程

```
用户: "订单表有哪些字段？"
   ↓
系统: 检索订单表 schema，注入 AI context
   ↓
AI: "订单表 (orders) 包含以下字段：
    - id (bigint, PK)
    - user_id (bigint, FK → users.id)
    - total_amount (decimal(10,2))
    - status (varchar(20))
    - created_at (datetime)"
   ↓
用户: "那 status 都有哪些值？"
   ↓
系统: 利用对话上下文 + CHECK 约束 / ENUM 信息
   ↓
AI: "status 字段的类型是 varchar(20)，从代码层面看一般包含：
    pending（待支付）、paid（已支付）、shipped（已发货）、
    completed（已完成）、cancelled（已取消）"
```

## 5. 技术要点

### SSE 流式响应

```
POST /api/projects/{id}/ask
Headers: Accept: text/event-stream
Body: { "question": "...", "session_id": "..." }

Response (SSE):
data: {"type": "chunk", "content": "订"}
data: {"type": "chunk", "content": "单"}
data: {"type": "chunk", "content": "表"}
data: {"type": "done"}
```

### Context 构造

为控制 token 消耗，按需构造上下文：

| 问题模式 | 注入上下文 |
|---------|-----------|
| 具体表的查询 | 仅该表 + 关联表 schema |
| 跨表关系 | 涉及的 2-3 张表 schema + 关系 |
| 全局搜索 | 所有表名 + 字段名（摘要模式） |
| 不明确 | 完整项目 schema（最多 30 张表摘要） |

### 上下文窗口管理

```
项目有 100 张表，全部注入将用完 context window。
策略：
1. 先用关键词匹配（表名/字段名）候选表
2. 仅注入候选表 + 其一/二度关联表的 schema
3. 超长时启用摘要模式：每张表仅保留 "表名 + PK + 关键字段"
```

## 6. 接口设计

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | /api/projects/{id}/ask | 提问（流式响应） |
| POST | /api/projects/{id}/ask/sync | 提问（非流式） |
| POST | /api/sessions | 创建对话会话 |
| GET | /api/sessions/{id}/history | 获取对话历史 |
| DELETE | /api/sessions/{id} | 删除会话 |

## 7. 验收标准

- [ ] "A 表和 B 表什么关系" — 正确回答关联字段和关系类型
- [ ] "包含 xxx 字段的表有哪些" — 列出所有匹配表，无遗漏
- [ ] "某某字段是什么意思" — 结合注释或字段名给出合理解释
- [ ] 提问响应首字到达时间 < 3s（SSE）
- [ ] 同一会话上下文保持连贯追问
- [ ] 项目超过 50 张表时，关键词检索命中率 > 95%
- [ ] 引用回答附带了表和字段名来源
